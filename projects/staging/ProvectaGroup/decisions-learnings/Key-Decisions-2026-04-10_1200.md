## Decisions

### D1 — Mobile compare chart: `<select>` dropdown over scrollable tabs
**Decision:** Replace the 7-tab horizontal strip with a styled `<select>` dropdown on mobile (≤768px). Tabs remain on desktop.
**Rationale:** 7 tabs cannot fit on a 375px viewport without forcing the user to scroll horizontally and discover hidden options. A dropdown natively exposes all 7 competitors at once, is familiar mobile UX, and requires no added dependencies.
**How to apply:** Any time a compare/tab UI has more than 4 options, default to a dropdown on mobile rather than a scrollable strip.

### D2 — `showPanel()` JS: use `data-panel` attributes, not `event.currentTarget`
**Decision:** Refactored `showPanel(id)` to find the active tab via `document.querySelector('.compare-tab[data-panel="${id}"]')` rather than `event.currentTarget`.
**Rationale:** `event.currentTarget` is coupled to the click event source. When the same function is called from a `<select>` onchange, `event.currentTarget` points to the select element (not a tab button), causing silent failures. Data attributes are source-agnostic.
**How to apply:** Whenever a JS function must work from multiple event sources (click + change + programmatic call), use data attributes for element identity — never rely on the implicit `event` global.

### D3 — Premier Realty: separate analysis document from proposal
**Decision:** Created `Premier_Realty_Market_Analysis_2026.docx` as a standalone document, leaving `Premier_Realty_Proposal_2026.docx` untouched.
**Rationale:** Analysis and proposal serve different engagement stages. Analysis is used for problem framing / credibility-building before the ask; proposal is used to close. Keeping them separate preserves optionality for the sales process.
**How to apply:** For all future prospects, maintain distinct "analysis" (problem-side) and "proposal" (solution-side) documents. Never conflate them.

### D4 — Netlify deployments: always copy new HTML files into deploy-pgco/
**Decision:** Any new HTML files intended for Netlify must be physically copied into the `deploy-pgco/` directory tree before running `npx netlify-cli deploy --prod`.
**Rationale:** Netlify CLI deploys only the files it finds in the specified `--dir`. Files elsewhere in the filesystem are invisible to it, causing 404s after deploy.
**How to apply:** Before every Netlify deploy, verify all expected files are under `deploy-pgco/`. Use `ls -R deploy-pgco/` to confirm.

### D5 — Relative back-links: use absolute URLs for cross-section links
**Decision:** The AI Memory Brain landing page back-link was changed from a relative path to `https://pgco.world`.
**Rationale:** Relative paths break when a page is accessed directly (e.g. from a search engine or shared link) rather than navigated to from the site root. Absolute URLs are always correct regardless of entry point.
**How to apply:** Any navigation link between major site sections should use absolute URLs, not relative paths.

### L2 — Chrome extension `file_upload` is blocked
The Claude-in-Chrome MCP `file_upload` tool is blocked by extension security policy ("Not allowed"). For Netlify deployments, always use the Netlify CLI via Bash sandbox — it is faster and more reliable anyway.

### L3 — CSS `@media` queries must be added before deployment
Large HTML files with no responsive CSS silently break on mobile. Any landing page built for desktop should have a responsive pass before going live. Minimum breakpoints: 1024px, 768px, 480px.

### L4 — `compare-col:first-child::after` for VS divider
Using a CSS `::after` pseudo-element on the first column card to inject the "vs" divider avoids adding extra HTML elements and keeps the markup clean. The divider inherits the section background to visually separate the two cards.
